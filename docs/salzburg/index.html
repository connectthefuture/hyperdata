<div class="full">
<h2 class="centre">XForms</h2>

<p>XForms originally designed as a replacement for HTML Forms.</p>
<ul>
<li>analysis of HTML features</li>
<li>requirements analysis derived from usage of HTML Forms and other
electronic forms systems.</li>
</ul>

<p>The resultant design</p>
<ul>
<li>MVC-based</li>
<li>intent-based controls</li>
<li>XML as a first-class data format, both for initialising data from
external sources, as for submission.</li>
</ul>
</div>

<div class="full">
<h2>Example</h2>

<p>What this concretely means is that the data is physically separated from the
controls in the form. The data is placed in the head of the document, and the
controls bind to the data.</p>
<pre>&lt;html xmlns="http://www.w3.org/1999/xhtml"&gt;
&lt;head&gt;
&lt;model xmlns="http://www.w3.org/2002/xforms"&gt;
&lt;instance&gt;
&lt;data xmlns=""&gt;
<strong>&lt;year&gt;2012&lt;/year&gt;...</strong>
&lt;/data&gt;
&lt;/instance&gt;
&lt;/model&gt;
&lt;/head&gt;
&lt;body&gt; ...</pre>

<p></p>
</div>

<div class="full">
<h2>Controls and initial values</h2>

<p>Controls in the body refer to values in the data instance(s) using XPath
expressions:</p>
<pre>&lt;input <strong>ref="year"</strong>&gt;...
&lt;input <strong>ref="event[1]/title/@language"</strong>&gt;...</pre>

<p>The controls can be initialised by putting values in the data:</p>
<pre>&lt;data xmlns=""&gt;<strong>&lt;year&gt;2001&lt;/year&gt;</strong>...&lt;/data&gt;</pre>

<p>but the data can also be initialised from external sources:</p>
<pre>&lt;instance <strong>src="http://www.example.org/events"</strong>/&gt;</pre>
</div>

<div>
<h2 class="centre">Constraints</h2>

<p>Relationships between, and restrictions on, values can be specified in the
model, allowing dependent values to be calculated automatically and data
checking to be performed on the client rather than on the server. </p>
<pre>&lt;bind nodeset="year" <strong>constraint</strong>=". &amp;gt; 1752"/&gt;<br />&lt;bind nodeset="state" <strong>required</strong>="../country = 'USA'"/&gt;<br />&lt;bind nodeset="age" <strong>calculate</strong>="../thisyear - ../birthdate/year"/&gt;
&lt;bind nodeset="birthdate" <strong>type</strong>="date"/&gt;</pre>
</div>

<div>
<h2>Output</h2>

<p>Values can be exposed in the document itself, using an output control: </p>
<pre>The result for the year &lt;output ref="year"/&gt; is ...</pre>
</div>

<h2>Intent-based Controls</h2>

<p>Controls are intent-based, by expressing what the control should do, rather
than how it should look. So a control like this:</p>
<pre>&lt;select1 ref="colour"&gt;
&lt;label&gt;Colour:&lt;/label&gt;
&lt;item&gt;&lt;label&gt;red&lt;/label&gt;
&lt;value&gt;#ff0000&lt;/value&gt;&lt;/item&gt;
&lt;item&gt;&lt;label&gt;green&lt;/label&gt;
&lt;value&gt;#00ff00&lt;/value&gt;&lt;/item&gt;
&lt;item&gt;&lt;label&gt;blue&lt;/label&gt;
&lt;value&gt;#0000ff&lt;/value&gt;&lt;/item&gt;
&lt;/select1&gt;</pre>

<p>can be represented in different ways depending purely on styling.</p>

<div>
<h2>Initial experience</h2>
<ul>
<li>far more powerful and flexible than the HTML Forms it was replacing</li>
<li>too slavishly followed the HTML design</li>
</ul>

<p>Particularly in the use of fixed strings rather than (potentially)
calculated values for such things as the submission URI.</p>

<p>As a consequence this restricted what was possible with the language.</p>
</div>

<div>
<h2>XForms 1.0 → 1.1</h2>

<p>As a consequence, XForms 1.1 addressed these shortcomings</p>

<p>Resultant language turned out to be far more than a forms language, but a
declarative application language. </p>

<p>Since XForms has input, output, and a processing engine, XForms is
Turing-complete, and much more than just forms is now possible with the
language.</p>
</div>

<div>
<h2>XForms 1.0 → 1.1</h2>

<p>Experience: application production time can be reduced by an order of
magnitude</p>

<p>One large project reporting a reduction from 5 years with 30 programmers
using traditional programming, to 1 year with 10 programmers using XForms.</p>
</div>

<div>
<h2>XForms 1.1 → XForms 2.0</h2>

<p>Biggest changes: XPath 2.0, AVTs</p>

<p>Another change: accept data in other formats than XML</p>
</div>

<div>
<h2>Data Opacity</h2>
<ul>
<li>XForms treats its data internally as if it is XML</li>
<li>XPath both to address data as to calculate new values</li>
<li>Not the intention that external data be only in XML</li>
</ul>
<ul>
<li>Just as a photo editor doesn't care about the external format, nor does
XForms</li>
<li>However, since the <em>internal</em> form of the data that XForms deals
with is XML (since the data is accessed using XPath), there has to be a
mapping between the external form and the internal one.</li>
</ul>
</div>

<div>
<h2>JSON</h2>

<p>An obvious data format widely in use on the web is JSON.</p>

<p>There are several mappings defined in both directions between XML and JSON,
but largely because JSON can only represent a subset of what XML can represent,
many of the mappings are cumbersome, and unnatural.</p>
</div>

<div>
<h2>Example</h2>

<p>For instance, just to take one example, here of the mapping from JXON, the
following XML:</p>
<pre>&lt;BOOKS&gt;
&lt;BOOK id="1"&gt;
&lt;TITLE&gt;My Favorite Book&lt;/TITLE&gt;
&lt;PRICE&gt;1.23&lt;/PRICE&gt;
&lt;/BOOK&gt;
&lt;BOOK id="1a"&gt;
&lt;TITLE&gt;XML for Dummies&lt;/TITLE&gt;
&lt;PRICE&gt;5.25&lt;/PRICE&gt;
&lt;/BOOK&gt;
&lt;BOOK id="3"&gt;
&lt;TITLE&gt;JSON for Dummies&lt;/TITLE&gt;
&lt;PRICE&gt;200.95&lt;/PRICE&gt;
&lt;/BOOK&gt;
&lt;/BOOKS&gt;</pre>

<p>would be transformed into:</p>
<pre>{
"childNodes": [
{
"childNodes": [
{
"childNodes": ["My Favorite Book"],
"tagName": "TITLE"
},
{
"childNodes": [1.23],
"tagName": "PRICE"
}
],
"id": 1,
"tagName": "BOOK"
},
{
"childNodes": [
{
"childNodes": ["XML for Dummies"],
"tagName": "TITLE"
},
{
"childNodes": [5.25],
"tagName": "PRICE"
}
],
"id": "1a",
"tagName": "BOOK"
},
{
"childNodes": [
{
"childNodes": ["JSON for Dummies"],
"tagName": "TITLE"
},
{
"childNodes": [200.95],
"tagName": "PRICE"
}
],
"id": 3,
"tagName": "BOOK"
}
],
"tagName": "BOOKS"
}</pre>

<div>
<h2>JSON in XForms</h2>

<p>During the design phase we went through several iterations</p>

<p>Key realisation: since the aim is only to address existing JSON stores, it
is not necessary to be able to convert every possible XML representation into
an equivalent JSON representation, only the reverse.</p>

<p>This reduces the task considerably, since it means several features of XML
do not have to be addressed, such as namespaces, attributes, and mixed
content.</p>
</div>

<div>
<h2>Requirements</h2>

<p>Some of the requirements for a mapping from JSON to XML for XForms
included:</p>
<ul>
<li>All possible JSON values be representable</li>
<li>Round-trippable, so that you can both read from and submit to a JSON
store.</li>
<li>As natural-looking selectors as possible.</li>
</ul>
</div>

<div>
<h2>Opaque data</h2>

<p>Ideally, an XForm processing JSON data shouldn't have to know which data
format has been used; so that, for instance, data such as</p>
<pre>{"company":"example.com", "locations":[{"city": "Amsterdam"},{"city": "London"}]}</pre>

<p>with the right mapping could be selected with XPath selectors like</p>
<pre>locations/city[1] </pre>

<p>In this way data could be loaded using content negotiation, and will work
whether the data comes in as XML or JSON.</p>
</div>

<div>
<h2>Transformation used</h2>

<p>The basic mapping designed is rather simple . Since JSON has no attributes,
all content can be represented in elements, and attributes are therefore free
to be used to help with the mapping.</p>

<p>Since a JSON value can have several values at the top level, a root element
is used <code>&lt;json&gt;</code>. JSON names become XML elements: </p>
<pre>{"name": "XForms"}</pre>

<p>becomes</p>
<pre>&lt;json&gt;&lt;name&gt;XForms&lt;/name&gt;&lt;/json&gt;</pre>
</div>

<div>
<h2>Types</h2>

<p>Strings are the default datatype. In order to allow the processor to
distinguish between {"size": 30} and {size: "30"} when serialising, other types
are marked:</p>
<pre>"age": 21</pre>

<p>becomes</p>
<pre>&lt;age type="integer"&gt;21&lt;/age&gt;</pre>

<p>and</p>
<pre>"registered": true</pre>

<p>becomes:</p>
<pre>&lt;registered type="boolean"&gt;true&lt;/registered&gt;</pre>
</div>

<div>
<h2>Nesting</h2>

<p>Nested values are obvious:</p>
<pre>"name": {"given": "Isaac", "family": "Newton"}</pre>

<p>becomes</p>
<pre>&lt;name&gt;&lt;given&gt;Isaac&lt;/given&gt;&lt;family&gt;Newton&lt;/family&gt;&lt;/name&gt;</pre>

<p></p>
</div>

<div>
<h2>Arrays</h2>

<p>Arrays are marked specially:</p>
<pre>"colour": ["red", "green", "blue"]</pre>

<p>becomes</p>
<pre>&lt;colour starts="array"&gt;red&lt;/colour&gt;
&lt;colour&gt;green&lt;/colour&gt;
&lt;colour&gt;blue&lt;/colour&gt;</pre>

<p>This allows selectors like <code>colour[3]</code> to work, but also allows
to distinguish things like single element arrays:</p>
<pre>{city: ["Amsterdam"]}</pre>

<p>from</p>
<pre>{city: "Amsterdam"}</pre>

<p>and empty arrays:</p>
<pre>{"set": []}</pre>

<p>from</p>
<pre>{"set": ""}</pre>
</div>

<div>
<h2>Example</h2>

<p>To take an example from the JSON site:</p>
<pre>{"bindings": [
{"ircEvent": "PRIVMSG",
"method": "newURI",
"regex": "^http://.*"},
{"ircEvent": "PRIVMSG",
"method": "deleteURI",
"regex": "^delete.*"},
{"ircEvent": "PRIVMSG",
"method": "randomURI",
"regex": "^random.*"}
]
}</pre>

<p>would become</p>
<pre>&lt;json&gt;
&lt;bindings starts="array"&gt;
&lt;ircEvent&gt;PRIVMSG&lt;/ircEvent&gt;
&lt;method&gt;newURI&lt;/method&gt;
&lt;regex&gt;^http://.*&lt;/regex&gt;
&lt;/bindings&gt;
&lt;bindings&gt;
&lt;ircEvent&gt;PRIVMSG&lt;/ircEvent&gt;
&lt;method&gt;deleteURI&lt;/method&gt;
&lt;regex&gt;^delete.*&lt;/regex&gt;
&lt;/bindings&gt;
&lt;bindings&gt;
&lt;ircEvent&gt;PRIVMSG&lt;/ircEvent&gt;
&lt;method&gt;randomURI&lt;/method&gt;
&lt;regex&gt;^random.*&lt;/regex&gt;
&lt;/bindings&gt;
&lt;/json&gt;</pre>

<p>and a JSON selector like</p>
<pre>bindings[0].method</pre>

<p>would become in XPath (JSON is 0-based, XPath 1-based):</p>
<pre>bindings[1]/method</pre>
</div>

<div>
<h2>Special Cases</h2>

<p>There are a small number of special cases that have to be accounted for:</p>
<ul>
<li>JSON allows the empty name "", which XML does not allow.</li>
<li>JSON names may contain characters that are not allowed as name characters
in XML.</li>
<li>JSON strings may contain any Unicode character; XML disallows most
characters below #x20.</li>
</ul>
</div>

<div>
<h2>Dealing with special cases</h2>

<p>Empty names and illegal name characters are easy to deal with: any character
that is not possible in XML is replaced with an underscore, and an attribute
<code>name</code> is added to the element giving the correct name. The empty
name is replaced with a single underscore, and an empty
<code>name</code> attribute is used.</p>

<p>For example: </p>
<pre>"$": "$"</pre>

<p>would be transcribed:</p>
<pre>&lt;_ name="$"&gt;$&lt;/_&gt;<br /></pre>
</div>

<div>
<h2>Characters</h2>

<p>The third is harder to deal with, with an example being:</p>
<pre>{"backspace": "\b"}</pre>

<p>The backspace character is completely disallowed in XML (even hex encoded),
leaving the only option to leave those illegal characters encoded in JSON
notation.</p>
</div>

<div>
<h2>Implementation</h2>

<p>Implementation of the mapping is relatively trivial: </p>

<p>At the point where an implementation normally receives a document of type
<code>application/xml</code> (or similar), either during initial instance
initialisation from an external resource, or as the return value of a
submission, if the media type of the resource is
<code>application/json<em>,</em></code> the resource can be parsed, and
transformed to an equivalent XML instance, as described above.</p>

<p>The media type can be recorded as an attribute of the root element, so that
it can be reused if the instance is to be resubmitted as JSON.</p>
</div>

<div>
<h2>Other formats</h2>

<p>Clearly this method can be extended to other datatypes such as VCARD <span
class="citation"></span> and iCalendar. For instance an iCalendar value such
as</p>
<pre>BEGIN:VCALENDAR
METHOD:PUBLISH
PRODID:-//Example/ExampleCalendarClient//EN
VERSION:2.0
BEGIN:VEVENT
ORGANIZER:mailto:a@example.com
DTSTART:19970701T200000Z
DTSTAMP:19970611T190000Z
SUMMARY:ST. PAUL SAINTS -VS- DULUTH-SUPERIOR DUKES
UID:0981234-1234234-23@example.com
END:VEVENT
END:VCALENDAR</pre>

<p>can be transformed to</p>
<pre>&lt;VCALENDAR&gt;
&lt;METHOD&gt;PUBLISH&lt;/METHOD&gt;
&lt;PRODID&gt;-//Example/ExampleCalendarClient//EN&lt;/PRODID&gt;
&lt;VERSION&gt;2.0&lt;/VERSION&gt;
&lt;VEVENT&gt;
&lt;ORGANIZER&gt;mailto:a@example.com&lt;/ORGANIZER&gt;
&lt;DTSTART&gt;19970701T200000Z&lt;/DTSTART&gt;
&lt;DTSTAMP&gt;19970611T190000Z&lt;/DTSTAMP&gt;
&lt;SUMMARY&gt;ST. PAUL SAINTS -VS- DULUTH-SUPERIOR DUKES&lt;/SUMMARY&gt;
&lt;UID&gt;0981234-1234234-23@example.com&lt;/UID&gt;
&lt;/VEVENT&gt;
&lt;/VCALENDAR&gt;</pre>

<p></p>
</div>

<div>
<h2>Conclusions</h2>

<p>Due to the lack of a need to represent arbitrary XML in JSON, dealing with
external JSON values in XForms becomes easy, and natural, in most cases not
even exposing the fact that the external data type is not XML in the XForm. The
approach can be extended to other types, and thanks to the generality of XML,
mostly without restriction.</p>

<p>Future XML: allow all Unicode please; and do something about character
entities...</p>
</div>
